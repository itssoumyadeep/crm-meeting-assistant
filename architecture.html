CRM Meeting Assistant Architecture

From transcript to CRM update, with a human approval gate.

This project combines a Streamlit front end, an ADK multi-agent pipeline, a read-only MCP server, and a local SQLite CRM store. The core rule is simple: the model can propose changes, but only a human can approve them.

Primary UI Streamlit workflow in src/ui/app.py
Analysis engine Sequential ADK pipeline in src/agents/pipeline.py
Custom skill Transcript handoff skill shapes summary and follow-up extraction
Data layer SQLite with WAL and column allow-lists
Safety model Prompt-injection scan, PII redaction, and stage validation

Architecture Summary

What each layer is responsible for.

Presentation

The Streamlit app exposes the full workflow in eight tabs: transcript input, analysis, summary, signals, sentiment, competitor mentions, review, and history.

Orchestration

The ADK pipeline first extracts transcript intelligence and then maps that analysis into CRM-relevant recommendations.

Safety

The only write-capable action is the guarded CRM proposal path. Invalid stages, injection strings, and sensitive identifiers are blocked before any write.

Persistence

SQLite stores contacts, deals, pending updates, and immutable audit logs. The app uses a local database for a fast demo loop.

Request Lifecycle

A single transcript moves through these stages before anything reaches the CRM.

Transcript capture

The user pastes or loads a sample transcript in the Streamlit UI.

Parallel analysis

The ADK agents extract a summary, action items, buying signals, competitor mentions, and sentiment.

CRM recommendation

The mapper agent proposes a stage and field updates, but does not write directly to the CRM.

Guardrail screening

Tool allow-lists, stage validation, and sensitive-data redaction happen before tool execution.

Human review

The proposed changes appear in the review tab for approve, edit, or reject actions.

Commit and audit

Approved changes update the SQLite CRM tables and are recorded in the audit trail.

Key Files

The architecture is split across these main code paths.

app/agent.py

ADK app entry point with the parallel analysis pipeline used by the deployment target.

app/fast_api_app.py

FastAPI and A2A server entry point for the agent runtime surface.

src/ui/app.py

Streamlit front end that runs the pipeline and presents review tabs.

Sequential agent chain, guardrails, PII redaction, and CRM proposal tool.

FastMCP server exposing read-only CRM lookups and stage options.

src/services/crm_service.py

Pending-update lifecycle, SQL allow-list enforcement, and audit log writes.

src/database/db_helper.py

SQLite connection helper with WAL mode and foreign-key support.

tests/unit/test_guardrails.py

Security-focused tests for injection blocking, redaction, and SQL allow-lists.

Design Notes

Why a sequential pipeline

Each stage enriches session state so downstream agents can build on prior results without reparsing the transcript.

Why a review queue

The app stages CRM changes in pending_updates so no model output reaches the CRM without human approval.

Why MCP is read-only

The agent can query contacts and valid deal stages without taking direct database dependencies into write paths.
